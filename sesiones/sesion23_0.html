Proposito de la sesion

Un programa no termina cuando compila: termina cuando puede evolucionar

  • Identificar deuda tecnica en codigo funcional: nombres confusos, funciones con multiples responsabilidades, logica duplicada.
  • Aplicar convenciones de calidad que hacen el codigo comprensible para quien lo modifique despues.
  • Relacionar cada decision tecnica con su costo de mantenimiento futuro.
  • Construir un plan de mejora modular basado en evidencia del codigo existente.

Concepto

Deuda tecnica: codigo que funciona pero que cuesta mantener

Definicion: La deuda tecnica es el costo adicional de trabajo que se genera cuando se elige una solucion rapida o descuidada en lugar de una solucion bien diseñada. Como una deuda financiera, si no se paga, crece con intereses.

Señales de deuda tecnica

  • Nombres de variables: x, temp2, aux.
  • Funciones de mas de 40 lineas.
  • Logica identica copiada en dos lugares.
  • Comentarios que explican "que hace" en vez de "por que".

Costo real

  • Cambiar una regla de negocio requiere modificar 4 funciones en vez de 1.
  • Un compañero no puede entender el codigo sin preguntarle al autor.
  • Un defecto en un lugar produce errores en otro lugar no relacionado visiblemente.

Analisis de codigo

Identificar deuda en un fragmento real

static void Procesar(int n) { double x = 0; for (int i = 1; i <= n; i++) { Console.Write("h: "); double h = double.Parse(Console.ReadLine()); Console.Write("t: "); double t = double.Parse(Console.ReadLine()); // calcula el neto (igual que la funcion de nomina) double neto = h * t; if (h > 40) neto += (h - 40) * t * 0.5; x += neto; Console.WriteLine($"Neto {i}: {neto:C}"); } Console.WriteLine($"Total: {x:C}"); }

Problemas identificados

  • Nombre x: no comunica su intencion. Deberia ser totalNeto.
  • Parametros h y t: sin contexto. Deberian ser horas y tarifa.
  • Logica duplicada: el calculo de neto ya existe en otra funcion; se esta copiando.
  • Mezcla UI + logica: leer, calcular y mostrar dentro del mismo ciclo.

Convenciones de calidad

Convencion 2: una funcion, una responsabilidad

Funcion con baja cohesion

Lee datos, valida, calcula Y muestra resultado en la misma funcion.

  • Si cambia el formato de salida, hay que editar la funcion de calculo.
  • No se puede probar el calculo sin la entrada interactiva.
  • Es imposible reutilizar solo la parte de calculo.

Funcion con alta cohesion

Cada funcion hace exactamente una cosa, bien definida en su nombre.

  • LeerDatos() → solo lee.
  • Validar() → solo valida.
  • Calcular() → solo calcula.
  • Mostrar() → solo presenta.

Pregunta de diagnostico: "¿Puedo describir lo que hace esta funcion en una sola frase sin usar 'y'?"

Convenciones de calidad

Convencion 3: reducir acoplamiento entre funciones

Alto acoplamiento (problematico)

static double valorGlobal = 0; static double Calcular() { valorGlobal = 100; // modifica global return valorGlobal * 2; }

Si Calcular modifica una variable global, cualquier otra funcion que la lea puede recibir un valor inesperado.

Bajo acoplamiento (correcto)

// Funcion pura: recibe todo por parametro static double Calcular(double valor) { return valor * 2; }

Solo depende de sus parametros. Se puede probar con cualquier valor sin preparar estado global.

Ciclo de vida

Las decisiones de hoy determinan el costo de mañana

Dato real de industria: entre el 40% y el 80% del costo total de un software se destina a mantenimiento (correcciones, nuevas reglas, cambios de plataforma). Un diseño modular reduce ese costo de forma directa.

Plan de mejora

Como construir un plan de mejora modular

Paso 1 — Inventario: listar todas las funciones del programa con su responsabilidad actual.
Paso 2 — Diagnosis: para cada funcion, responder: ¿hace mas de una cosa? ¿usa variables globales? ¿esta duplicada?
Paso 3 — Priorizar: atacar primero la deuda que mas impacta en el siguiente cambio previsto.
Paso 4 — Mejorar con pruebas: antes de refactorizar, construir al menos un caso de prueba que confirme que la funcion ya funciona. Refactorizar sin romper ese caso.

Refactorizacion segura

Ejemplo: extraer logica duplicada a una funcion

Antes (con deuda)

// En funcion A: double neto = horas * tarifa; if (horas > 40) neto += (horas - 40) * tarifa * 0.5; // En funcion B (copia exacta): double neto = horas * tarifa; if (horas > 40) neto += (horas - 40) * tarifa * 0.5;

Despues (mejorado)

// Nueva funcion extraida: static double CalcularNeto( double horas, double tarifa) { double neto = horas * tarifa; if (horas > 40) neto += (horas - 40) * tarifa * 0.5; return neto; } // Funcion A y B usan la misma funcion: double neto = CalcularNeto(h, t);

El caso de prueba de la funcion original debe seguir pasando despues de la refactorizacion.

Convenciones de calidad

Convencion 4: documentar el por que, no el que

Comentario que no aporta

// multiplica horas por tarifa double neto = horas * tarifa;

El codigo ya dice lo mismo. El comentario es ruido.

Comentario que aporta

// Se retorna porcentaje (0.20) y no monto // para que el precio base pueda cambiar // sin modificar esta funcion. return 0.20;

Explica la decision de diseño que no es obvia leyendo el codigo.

Regla: comentar decisiones no evidentes, restricciones del negocio y contratos que la firma no puede expresar.

Control de versiones

El commit como registro de decision tecnica

Mensajes sin valor

  • cambios
  • fix
  • actualizacion del lunes
  • no se que mas agregar

Estos mensajes no permiten entender que cambio ni por que.

Mensajes con valor tecnico

  • refactor: extraer CalcularNeto de Procesar()
  • fix: corregir conteo de errores en ProcesarLote
  • feat: agregar validacion de tarifa negativa
  • docs: agregar comentario de contrato en CalcularDescuento

Formato sugerido: tipo: descripcion breve del cambio (tipos: feat, fix, refactor, docs, test).

Practica guiada

Aplicar una mejora de calidad

Reglas del proceso

  • Antes de cambiar cualquier linea, ejecutar el proyecto y anotar la salida esperada.
  • Hacer solo un cambio a la vez (renombrar OR extraer OR separar).
  • Compilar y verificar que la salida sigue siendo la misma.
  • Hacer commit con mensaje descriptivo del tipo de mejora.

Mejoras sugeridas para comenzar

  • Renombrar una variable con nombre generico a nombre semantico.
  • Agregar un comentario de contrato en una funcion que tiene restricciones no obvias.
  • Separar un bloque de consola que esta mezclado con un calculo.

Cierre de sesion

Codigo de calidad no es perfeccion: es codigo que puede evolucionar

Lo que construimos hoy

  • Identificamos deuda tecnica con criterios concretos.
