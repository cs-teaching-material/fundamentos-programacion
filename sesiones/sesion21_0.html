Proposito de la sesion

Hacer que el codigo modular diga lo que hace

  • Reconocer senales de codigo que funciona pero que es dificil de leer o mantener (code smells).
  • Aplicar tres reglas concretas de calidad: nombres significativos, funcion de una sola responsabilidad y funciones cortas.
  • Entender que es la sobrecarga de metodos en C# y cuando usarla aporta claridad en lugar de confusion.
  • Refactorizar el sistema de la sesion anterior sin cambiar su comportamiento externo.
  • Distinguir entre codigo que funciona y codigo de calidad como diferencia profesional.

Punto de partida: El sistema de nomina de la sesion 20 funciona correctamente. Hoy lo vamos a mejorar sin tocar su logica: solo su expresividad.

Sesion 21 · Contexto3 / 20
Concepto clave

Code smell: cuando el codigo huele mal sin estar roto

Definicion: Un code smell es una senal en el codigo que indica un problema de diseno o legibilidad, aunque el programa compile y ejecute sin errores. No es un bug: es una advertencia de que algo puede mejorar.

Analogia

Una habitacion puede estar "ordenada" si todo fue metido bajo la cama. Funcionalmente nada se perdio, pero si alguien mas necesita buscar algo, tendra un problema. El codigo con code smells es igual.

Por que importa en U3

Construimos funciones para que el codigo sea mas manejable. Si las funciones tienen nombres vagos o hacen demasiadas cosas, perdemos el beneficio de haberlas separado.

Origen del termino: Martin Fowler, Refactoring: Improving the Design of Existing Code (1999). Hoy es vocabulario estandar en la industria.

Sesion 21 · Los tres smells5 / 20
Regla de calidad 1

Los nombres deben revelar la intencion

Evitar
static double Calc(double a, double b) { return a * b; } double res = Calc(horas, tarifa); double r2 = Calc(res, 0.92);
Preferir
static double CalcularSalarioBruto( double horasTrabajadas, double tarifaPorHora) { return horasTrabajadas * tarifaPorHora; } double salarioBruto = CalcularSalarioBruto(h, t); double pagoFinal = AplicarDeduccionLegal(salarioBruto);

Regla practica: si necesitas un comentario para explicar que hace una variable o funcion, el nombre no es suficientemente bueno.

Sesion 21 · Regla 1: Nombres6 / 20
Regla de calidad 2

Una funcion = una responsabilidad (Principio SRP)

Funcion que hace tres cosas
static void ProcesarYMostrar(int idx) { // Responsabilidad 1: leer datos Console.Write("Horas: "); double h = double.Parse(Console.ReadLine()); // Responsabilidad 2: calcular double neto = h * 15000 * 0.92; // Responsabilidad 3: mostrar Console.WriteLine($"Empleado {idx}: {neto:C}"); }

El problema concreto

Si la empresa cambia la forma de ingresar datos (archivo CSV en vez de consola), hay que reescribir una funcion que tambien calcula y muestra. Tres razones para cambiarla = tres fuentes de bugs.

Prueba rapida: describe la funcion en una oracion. Si necesitas la palabra "y" mas de una vez, tiene mas de una responsabilidad.

Sesion 21 · Regla 2: SRP7 / 20
Regla de calidad 3

Una funcion debe caber en la pantalla sin hacer scroll

¿Por que 20 lineas como limite orientativo?

  • La memoria de trabajo humana retiene entre 5 y 9 elementos simultaneos.
  • Una funcion larga obliga a desplazarse y perder el contexto visual.
  • Si supera las 20 lineas, casi siempre hay un subproblema dentro que puede tener su propio nombre.

No es una regla dogmatica: 25 lineas con comentarios claros pueden ser mejores que 15 lineas confusas.

Senales de que hay que extraer una subfuncion

  • Hay un bloque con un comentario que lo describe (ese comentario deberia ser el nombre de una funcion).
  • Un bloque if tiene mas de 8 lineas internas.
  • Se repite la misma logica en dos lugares con variables diferentes (DRY).

Sesion 21 · Diagnostico S209 / 20
Concepto nuevo

Sobrecarga de metodos: el mismo nombre, distintas firmas

Definicion: La sobrecarga (method overloading) permite definir dos o mas metodos con el mismo nombre en la misma clase, siempre que difieran en la cantidad o el tipo de sus parametros. El compilador elige cual llamar segun los argumentos que recibe.

// Version 1: horas normales (sin extras) static double CalcularSalarioBruto( double horas, double tarifa) { return horas * tarifa; } // Version 2: con recargo por horas extra static double CalcularSalarioBruto( double horasNormales, double horasExtra, double tarifa) { double base = horasNormales * tarifa; double extra = horasExtra * tarifa * 1.5; return base + extra; }

Que gana el codigo

  • El nombre expresa el proposito sin sufijos artificiales como CalcularSalarioBrutoConExtra.
  • El sitio de llamada es legible: quien llama con 3 argumentos activa automaticamente la logica de extras.
  • Las dos versiones comparten semantica: ambas calculan salario bruto, solo difieren en la informacion disponible.
Sesion 21 · Sobrecarga: definicion10 / 20
Mecanismo

Como elige el compilador cual version ejecutar

Resolucion en tiempo de compilacion

El compilador examina los argumentos en el sitio de llamada y busca la firma que coincida. Si hay ambiguedad, lanza un error de compilacion; no elige arbitrariamente.

// Llama a la version de 2 parametros: double s1 = CalcularSalarioBruto(40, 15000); // Llama a la version de 3 parametros: double s2 = CalcularSalarioBruto(40, 8, 15000); // Error: no hay version con 1 parametro: double s3 = CalcularSalarioBruto(40); // no compila

Reglas de diferenciacion validas

  • Distintos tipos de parametros: int vs double → valido.
  • Distinta cantidad de parametros → valido.
  • Solo distinto nombre de parametro → NO es sobrecarga, es error.
  • Solo distinto tipo de retorno → NO es sobrecarga valida.
Sesion 21 · Resolucion de sobrecarga11 / 20
Criterio de diseno

Sobrecarga: cuando ayuda y cuando confunde

Usar sobrecarga cuando...
  • Las versiones realizan la misma operacion conceptual con diferente informacion disponible.
  • El nombre es igualmente valido para todas las versiones sin ambiguedad.
  • Ejemplo: CalcularSalarioBruto con y sin horas extra.
  • Ejemplo: Mostrar(nombre) y Mostrar(nombre, cargo).
Evitar sobrecarga cuando...
  • Las versiones hacen cosas semanticamente distintas aunque relacionadas.
  • El lector no puede deducir cual version se esta llamando sin leer la firma completa.
  • Se usa solo para evitar escribir dos nombres distintos.
  • Ejemplo confuso: Calcular(horas, tarifa) y Calcular(sueldo, porcentaje) con logicas distintas.

Regla de oro: si al leer la llamada no queda claro cual sobrecarga se invoca, la sobrecarga no deberia existir.

Sesion 21 · Criterio de sobrecarga12 / 20
Comparacion de tecnicas

Sobrecarga vs parametros con valor por defecto

Parametro con valor por defecto

static double CalcularSalarioBruto( double horas, double tarifa, double horasExtra = 0) { double base = horas * tarifa; double extra = horasExtra * tarifa * 1.5; return base + extra; } // Ambas llamadas son validas: CalcularSalarioBruto(40, 15000); CalcularSalarioBruto(40, 15000, 8);

Una sola firma. Si no se pasa horasExtra, vale 0 y el recargo es cero.

Cuando preferir cada opcion

  • Parametro por defecto: cuando la logica interna es identica y el parametro solo agrega datos. Menos duplicacion.
  • Sobrecarga: cuando la logica interna difiere segun los datos, o cuando el valor por defecto seria semanticamente confuso.

En el dominio laboral, "0 horas extra" y "no aplica horas extra" son conceptos distintos. La sobrecarga los separa con mayor precision.

Practica guiada

Paso 1: extraer la lectura de datos a su propia funcion

Antes: responsabilidades mezcladas
static bool TryProcesarEmpleado(int idx, out double neto, out double pago, out string err) { // Leer datos aqui dentro Console.Write("Horas: "); if (!double.TryParse(Console.ReadLine(), out double h)) { neto=pago=0; err="Horas invalidas"; return false; } Console.Write("Tarifa: "); if (!double.TryParse(Console.ReadLine(), out double t)) { neto=pago=0; err="Tarifa invalida"; return false; } // ...luego valida y calcula en el mismo cuerpo }
Despues: responsabilidades separadas
static bool LeerDatosEmpleado(int idx, out double horas, out double tarifa) { Console.Write($"Empleado {idx} - Horas: "); if (!double.TryParse(Console.ReadLine(), out horas)) { tarifa = 0; return false; } Console.Write($"Empleado {idx} - Tarifa: "); return double.TryParse(Console.ReadLine(), out tarifa); } // TryProcesarEmpleado ahora solo valida y calcula: static bool TryProcesarEmpleado(int idx, out double neto, out double pago, out string err) { if (!LeerDatosEmpleado(idx, out double h, out double t)) { neto=pago=0; err="Datos no legibles"; return false; } // ...continua con la logica de calculo }

Practica guiada

Paso 2: renombrar funciones y reducir el numero de parametros

MostrarConsolidado con 5 parametros
// Quien lee esto no sabe si el orden es correcto MostrarConsolidado(n, exitos, errores, totalNeto, totalPago);

5 parametros del mismo tipo: facil invertir dos sin que el compilador se queje.

Con un tipo de resumen
// Estructura que agrupa los datos del lote struct ResumenLote { public int Total, Exitos, Errores; public double TotalBruto, TotalNeto; } ResumenLote resumen = new ResumenLote(); // ...se rellena dentro del ciclo... MostrarResumenLote(resumen);

Un solo parametro con nombre expresivo. El orden ya no puede equivocarse.

Nota: struct es un tipo de valor en C#. Es apropiado para datos simples sin comportamiento, como este resumen. Se profundiza en cursos posteriores.

Cierre de sesion

Codigo que funciona y codigo de calidad no son lo mismo

Lo que construimos hoy

  • Identificamos code smells: nombres vagos, SRP violado, funciones largas.
  • Aplicamos tres reglas de calidad concretas al sistema de la S20.
  • Aprendimos la sobrecarga de metodos: misma operacion, distinta informacion disponible.
  • Distinguimos cuando sobrecargar aporta claridad y cuando genera confusion.
