Concepto fundamental

La clase es el molde; el objeto es el pastel

Analogia: Imagina un molde de silicona con forma de estrella. El molde no tiene sabor, no tiene color, no puedes comerlo. Pero cada vez que viertes masa de diferente sabor y color obtienes un pastel distinto: uno de chocolate amargo con glaseado rojo, otro de vainilla con chispas doradas. Todos tienen la misma forma de estrella porque salieron del mismo molde, pero cada uno es unico e independiente.

La clase (el molde de silicona)

  • Existe una sola vez en el codigo.
  • Define que caracteristicas y acciones tendra cada objeto.
  • No ocupa espacio util hasta que se hornea un pastel.
  • Puedes hacer infinitos objetos con el mismo molde.

El objeto (el pastel horneado)

  • Nace con new — es el momento de "hornear".
  • Cada pastel tiene su propio sabor y color independiente.
  • Cambiar el relleno de un pastel no afecta a los demas.
  • Vive en la memoria mientras el programa lo necesita.

Sintaxis basica

Declarar una clase y crear objetos en C#

// La clase es el molde — se define una sola vez class Empleado { // Campos: las caracteristicas de cada pastel public string Nombre; public double SalarioBase; public int Horas; // Metodo: lo que el empleado sabe hacer public double CalcularPago() { return SalarioBase * Horas; } } class Program { static void Main() { // Dos pasteles distintos del mismo molde Empleado e1 = new Empleado(); // primer horneado e1.Nombre = "Ana"; e1.SalarioBase = 12000; e1.Horas = 160; Empleado e2 = new Empleado(); // segundo horneado e2.Nombre = "Luis"; e2.SalarioBase = 15000; e2.Horas = 140; Console.WriteLine(e1.CalcularPago()); // 1920000 Console.WriteLine(e2.CalcularPago()); // 2100000 } }

Cada objeto es independiente

Cambiar e1.Horas = 80 no afecta a e2.Horas en lo absoluto. Son dos pasteles horneados por separado: aunque salieron del mismo molde, modificar la decoracion de uno no altera al otro.

La clase Empleado es el plano de arquitectura. e1 y e2 son las dos casas construidas con ese plano: misma distribucion, distintos propietarios.

Encapsulamiento

Por que no dejar todos los valores expuestos: la vitrina vs la caja fuerte

Analogia: Imagina una joyeria. Los anillos exhibidos en la vitrina los puede ver cualquiera — eso es public. Pero el cofre donde la joyera guarda los diamantes sin tallar esta bajo llave — eso es private. La joyera decide exactamente que se muestra, a que precio y bajo que condiciones. Sin ese control, cualquiera podria tomar un diamante y ponerle el precio que quiera.

Sin encapsulamiento (la vitrina sin cristal)

Empleado e = new Empleado(); e.SalarioBase = -5000; // invalido! e.Horas = -10; // invalido! // Cualquiera pone el precio que quiere // y el objeto queda en estado corrupto

Con encapsulamiento (vitrina con cristal y cerradura)

Empleado e = new Empleado(); // La propiedad valida antes de aceptar e.SalarioBase = -5000; // Lanza ArgumentException automaticamente // El diamante nunca sale de la caja fuerte // con un precio incorrecto

Propiedades

Propiedades: la ventanilla del banco que controla cada transaccion

class Empleado { // Campo privado: el cofre bajo llave private double _salarioBase; private int _horas; // Propiedad: la ventanilla con su guardia public double SalarioBase { get { return _salarioBase; } set { if (value <= 0) throw new ArgumentException( "Salario debe ser positivo"); _salarioBase = value; } } public int Horas { get { return _horas; } set { if (value < 0 || value > 744) throw new ArgumentException( "Horas fuera de rango"); _horas = value; } } // Propiedad auto-implementada: el compilador // genera el campo privado automaticamente public string Nombre { get; set; } public double CalcularPago() => _salarioBase * _horas; } class Program { static void Main() { Empleado e = new Empleado(); e.Nombre = "Ana"; e.SalarioBase = 12000; // pasa la validacion e.Horas = 160; Console.WriteLine(e.CalcularPago()); // 1920000 // Esto lanza ArgumentException: "Salario debe ser positivo" // e.SalarioBase = -5000; } }

Como funciona la ventanilla

  • get: alguien pide ver el valor — la ventanilla lo entrega.
  • set: alguien quiere cambiar el valor — el guardia (value) lo revisa primero.
  • Si el valor es invalido, el guardia rechaza la transaccion antes de que llegue al cofre.
  • El cofre interno (_salarioBase) jamas queda en estado corrupto.

value es la palabra reservada que contiene lo que alguien intenta asignar.

Constructor

El constructor: el ritual de inauguracion que le da vida al objeto

Analogia: Cuando un restaurante abre por primera vez, antes de recibir clientes el dueno hace un ritual de inauguracion: enciende las luces, coloca el menu en cada mesa, verifica que la caja registradora tenga cambio y pone el letrero de "Abierto". Si algo falla en ese ritual — por ejemplo no hay caja — el restaurante simplemente no abre. El constructor es ese ritual: garantiza que el objeto abre sus puertas en condiciones perfectas o no abre en absoluto.

class Empleado { private double _salarioBase; private int _horas; public string Nombre { get; private set; } // Constructor: mismo nombre que la clase, sin tipo de retorno public Empleado(string nombre, double salarioBase, int horas) { // Si algo falla, el objeto NO se crea (no abre el restaurante) if (salarioBase <= 0) throw new ArgumentException("Salario invalido"); if (horas < 0) throw new ArgumentException("Horas invalidas"); Nombre = nombre; _salarioBase = salarioBase; _horas = horas; } public double CalcularPago() => _salarioBase * _horas; public override string ToString() => $"{Nombre} | pago: {_salarioBase * _horas:C}"; } class Program { static void Main() { // El restaurante abre solo si todo esta correcto Empleado e = new Empleado("Ana", 12000, 160); Console.WriteLine(e); // Ana | pago: $1,920,000.00 // Este intento lanza excepcion: el restaurante no abre // Empleado mal = new Empleado("X", -1, 160); // ArgumentException } }

Miembros estaticos

Instancia vs estatico: el apodo propio vs el apellido familiar

Analogia: En la familia Garcia cada hijo tiene su apodo propio — Juan es "Juancho", Maria es "Mafe" — ese apodo es de instancia: es personal, nadie mas lo tiene. Pero el apellido "Garcia" es de toda la familia: es estatico, compartido por todos, y lo accedes por el apellido directamente, no por cada persona individualmente.

class Empleado { // Estatico: el "apellido" compartido por todos los objetos public static int TotalEmpleados = 0; public string Nombre { get; private set; } public Empleado(string nombre) { Nombre = nombre; TotalEmpleados++; // cada nuevo objeto actualiza el contador familiar } } class Program { static void Main() { Empleado a = new Empleado("Ana"); Empleado b = new Empleado("Luis"); Empleado c = new Empleado("Mia"); // Se accede por el nombre de la clase, no por el objeto Console.WriteLine(Empleado.TotalEmpleados); // 3 // a.Nombre es "Ana" — apodo personal (instancia) // Empleado.TotalEmpleados es 3 — apellido familiar (estatico) } }

Buenas practicas

Sobreescribir ToString: la etiqueta en la mermelada

Analogia: Tienes tres frascos de mermelada en la despensa. Si ninguno tiene etiqueta, al abrirlos no sabes si son de fresa, de mango o de ciruela. Si cada frasco tiene su etiqueta, de un vistazo sabes exactamente que hay adentro. ToString() es la etiqueta que pones en el objeto para que al mostrarlo en pantalla diga algo util — no el nombre tecnico del frasco.

class Empleado { public string Nombre { get; private set; } public double SalarioBase { get; private set; } public int Horas { get; private set; } public Empleado(string nombre, double salarioBase, int horas) { Nombre = nombre; SalarioBase = salarioBase; Horas = horas; } public double CalcularPago() => SalarioBase * Horas; // Sin override: Console.WriteLine(e) imprime "Empleado" — frasco sin etiqueta // Con override: imprime la etiqueta util que nosotros definimos public override string ToString() => $"[{Nombre}] pago mensual: {CalcularPago():C}"; } class Program { static void Main() { Empleado e1 = new Empleado("Ana", 12000, 160); Empleado e2 = new Empleado("Luis", 15000, 140); Console.WriteLine(e1); // [Ana] pago mensual: $1,920,000.00 Console.WriteLine(e2); // [Luis] pago mensual: $2,100,000.00 } }

Colecciones de objetos

Arreglo de objetos: el cajón con varios frascos etiquetados

using System; class Empleado { public string Nombre { get; private set; } public double SalarioBase { get; private set; } public int Horas { get; private set; } public Empleado(string nombre, double salarioBase, int horas) { Nombre = nombre; SalarioBase = salarioBase; Horas = horas; } public double CalcularPago() => SalarioBase * Horas; public override string ToString() => $"{Nombre,-12} | {CalcularPago(),14:C}"; } class Program { static void Main() { // El cajon: tres frascos distintos, todos del mismo molde Empleado[] equipo = { new Empleado("Ana", 12000, 160), new Empleado("Luis", 15000, 140), new Empleado("Mia", 11000, 168), }; double totalNomina = 0; foreach (Empleado e in equipo) { Console.WriteLine(e); // usa ToString() totalNomina += e.CalcularPago(); } Console.WriteLine($"\nTotal nomina: {totalNomina:C}"); } } // Ana | $1,920,000.00 // Luis | $2,100,000.00 // Mia | $1,848,000.00 // Total nomina: $5,868,000.00

Organizacion del codigo

Cada clase en su propio archivo: un cuarto para cada cosa

Estructura recomendada

// ── Empleado.cs ────────────────────── class Empleado { public string Nombre { get; private set; } public double SalarioBase { get; private set; } public int Horas { get; private set; } public Empleado(string n, double s, int h) { Nombre = n; SalarioBase = s; Horas = h; } public double CalcularPago() => SalarioBase * Horas; public override string ToString() => $"{Nombre,-12} | {CalcularPago():C}"; } // ── Program.cs ─────────────────────── class Program { static void Main() { // Solo orquestacion y entrada/salida Empleado e = new Empleado("Ana", 12000, 160); Console.WriteLine(e); } }

Por que separar — la analogia de la cocina

En un restaurante, la receta del chef (como se prepara cada plato) vive en la cocina. El menu que ve el cliente (como se presenta) vive en el salon. Si cambia la receta, solo va al chef. Si cambia el menu, solo se imprime de nuevo. Nadie confunde los dos mundos.

  • Cambias la regla de pago → solo editas Empleado.cs.
  • Cambias como se muestra → solo editas Program.cs.
  • Empleado.cs puede reutilizarse en otro proyecto sin tocar nada mas.

Ejemplo integrador

Clase CuentaBancaria: encapsulamiento en accion

class CuentaBancaria { private double _saldo; public string Titular { get; private set; } public double Saldo => _saldo; // ventanilla de solo lectura public CuentaBancaria(string titular, double saldoInicial) { if (saldoInicial < 0) throw new ArgumentException("Saldo inicial no puede ser negativo"); Titular = titular; _saldo = saldoInicial; } public void Depositar(double monto) { if (monto <= 0) throw new ArgumentException("Monto invalido"); _saldo += monto; } public void Retirar(double monto) { if (monto <= 0 || monto > _saldo) throw new ArgumentException("Retiro no valido"); _saldo -= monto; } public override string ToString() => $"[{Titular}] saldo actual: {_saldo:C}"; } class Program { static void Main() { CuentaBancaria c = new CuentaBancaria("Ana", 500000); c.Depositar(200000); // _saldo = 700 000 c.Retirar(100000); // _saldo = 600 000 Console.WriteLine(c); // [Ana] saldo actual: $600,000.00 // c._saldo = 9999; // ERROR: campo privado, no accesible } }

Modificadores de acceso

public, private, protected: el plano de la casa y sus habitaciones

Analogia: public es la sala de estar — cualquier visita entra. private es el cuarto del dueno bajo llave — solo el vive ahi, nadie mas entra. protected es el cuarto de juegos familiar — los hijos (subclases) pueden entrar, pero los vecinos no. Lo veremos en herencia, sesion 25.

ModificadorQuien puede accederUso tipico
publicCualquier codigo, desde cualquier archivo.Propiedades y metodos que forman la interfaz visible del objeto.
privateUnicamente el codigo dentro de la misma clase.Campos internos y metodos auxiliares que no deben exponerse.
protectedLa clase y sus clases hijas (subclases).Campos que las subclases necesitan pero el mundo exterior no debe tocar.

Regla de oro: empieza siempre con private. Solo lo abres a public cuando hay una razon clara para que codigo externo lo use.

Solucion practica

Clase Producto: implementacion y uso completos

class Producto { private string _nombre; private double _precio; private int _stock; public string Nombre => _nombre; public double Precio => _precio; public int Stock => _stock; public Producto(string nombre, double precio, int stock) { if (precio <= 0) throw new ArgumentException("Precio invalido"); if (stock < 0) throw new ArgumentException("Stock invalido"); _nombre = nombre; _precio = precio; _stock = stock; } public void Vender(int cantidad) { if (cantidad <= 0 || cantidad > _stock) throw new InvalidOperationException("Stock insuficiente"); _stock -= cantidad; } public void Reabastecer(int cantidad) { if (cantidad <= 0) throw new ArgumentException("Cantidad invalida"); _stock += cantidad; } public override string ToString() => $"{_nombre,-15} | precio: {_precio,10:C} | stock: {_stock,4}"; } class Program { static void Main() { Producto p1 = new Producto("Laptop", 3500000, 10); Producto p2 = new Producto("Teclado", 85000, 50); Producto p3 = new Producto("Mouse", 45000, 30); p1.Vender(3); p2.Vender(12); p3.Reabastecer(20); Console.WriteLine(p1); // Laptop | precio: $3,500,000 | stock: 7 Console.WriteLine(p2); // Teclado | precio: $85,000 | stock: 38 Console.WriteLine(p3); // Mouse | precio: $45,000 | stock: 50 } }

Cierre de sesion

Lo que construiste hoy

Conceptos nuevos

  • Clase: el molde — define la forma de todos los objetos de ese tipo.
  • Objeto: el pastel horneado — instancia real creada con new.
  • Encapsulamiento: campos privados + propiedades que validan antes de guardar.
  • Constructor: garantiza que el objeto abre sus puertas en estado valido.
  • Estatico: el apellido familiar — compartido por todos los objetos.

Que viene en sesion 25

  • Herencia: el hijo recibe todo del padre y agrega lo suyo.
  • Polimorfismo: el mismo mensaje, respuestas distintas segun el tipo.
  • Clases abstractas: planos que no pueden construirse directamente.
  • Interfaces: contratos de comportamiento que cualquier clase puede firmar.
