Concepto fundamental

La clase es el formulario de contratacion; el objeto es la ficha del empleado

Analogia: El departamento de Recursos Humanos tiene un formulario de contratacion estandar: nombre, puesto, salario, fecha de ingreso. El formulario no es ningun empleado — es solo la estructura en papel. Cuando contratan a Ana lo llenan con sus datos, cuando contratan a Luis lo llenan con los suyos. Cada ficha resultante es una persona real e independiente, aunque ambas vienen del mismo formulario.

La clase (el formulario de contratacion)

  • Existe una sola vez en el codigo.
  • Define que datos y acciones tendra cada empleado.
  • No es ningun empleado real hasta que alguien lo llena.
  • Puedes contratar infinitos empleados con el mismo formulario.

El objeto (la ficha del empleado)

  • Nace con new — es el momento de "contratar".
  • Cada ficha tiene sus propios datos independientes.
  • Cambiar el salario de Ana no afecta el salario de Luis.
  • Vive en la memoria mientras el programa lo necesita.

Sintaxis basica

Declarar una clase y crear objetos en C#

// La clase es el formulario de contratacion — se define una sola vez class Empleado { // Campos: los datos de cada empleado contratado public string Nombre; public double SalarioBase; public int Horas; // Metodo: lo que el empleado sabe hacer public double CalcularPago() { return SalarioBase * Horas; } } class Program { static void Main() { // Dos fichas distintas del mismo formulario de contratacion Empleado e1 = new Empleado(); // primera contratacion e1.Nombre = "Ana"; e1.SalarioBase = 12000; e1.Horas = 160; Empleado e2 = new Empleado(); // segunda contratacion e2.Nombre = "Luis"; e2.SalarioBase = 15000; e2.Horas = 140; Console.WriteLine(e1.CalcularPago()); // 1920000 Console.WriteLine(e2.CalcularPago()); // 2100000 } }

Cada objeto es independiente

Cambiar e1.Horas = 80 no afecta a e2.Horas en lo absoluto. Son dos fichas de empleado completadas por separado: aunque nacieron del mismo formulario, los datos de una no modifican los de la otra.

La clase Empleado es el formulario en blanco de RRHH. e1 y e2 son las dos fichas completadas: mismo formato, datos distintos. Cambiar e1.Horas = 80 no afecta a e2.Horas en lo absoluto.

Encapsulamiento

El problema: cuando los datos quedan expuestos sin control

Conexion con lo que ya sabes: Hasta ahora hemos usado variables sueltas. Con clases, esas variables viven dentro del objeto. Si las dejamos public, cualquier parte del programa puede cambiarlas con cualquier valor — incluso valores que no tienen sentido. El encapsulamiento es la tecnica para evitarlo: guardas los datos adentro y solo permites cambiarlos de forma controlada.

Sin control — cualquiera escribe lo que quiere

Empleado e = new Empleado(); e.Nombre = "Ana"; e.SalarioBase = -5000; // no tiene sentido e.Horas = -10; // tampoco // El programa sigue sin quejarse... // pero el calculo dara un resultado // completamente incorrecto. Console.WriteLine(e.CalcularPago()); // 50000 ???

Con control — la clase cuida sus datos

Empleado e = new Empleado(); e.Nombre = "Ana"; // SalarioBase y Horas ahora son // propiedades con logica de validacion. // Veremos como escribirlas en el // siguiente slide. e.SalarioBase = 12000; // acepta: es valido e.Horas = 160; // acepta: es valido

La clase decide que valores acepta. El codigo externo no puede saltarse esa decision.

Propiedades

Campos privados y propiedades: como la clase protege sus datos

class Empleado { // Campo privado: solo la clase puede tocarlo private double _salarioBase; // Propiedad: la "ventanilla" para entrar y salir public double SalarioBase { get { return _salarioBase; } set { if (value > 0) // solo acepta si es valido _salarioBase = value; // si no, simplemente no cambia nada } } public string Nombre; // este lo dejamos publico por ahora public double CalcularPago() { return _salarioBase * 160; } } class Program { static void Main() { Empleado e = new Empleado(); e.Nombre = "Ana"; e.SalarioBase = 12000; // acepta: 12000 > 0 e.SalarioBase = -5000; // ignora: -5000 no pasa el if Console.WriteLine(e.SalarioBase); // 12000 (no cambio) Console.WriteLine(e.CalcularPago()); // 1920000 } }

Como funciona paso a paso

  • private double _salarioBase — el campo vive adentro de la clase, nadie externo puede tocarlo directamente.
  • get — cuando alguien lee SalarioBase, este bloque devuelve el valor.
  • set — cuando alguien escribe SalarioBase = algo, este bloque recibe ese "algo" en la palabra reservada value y decide si lo acepta.
  • Si value no pasa la condicion, el campo interno no cambia — el objeto queda intacto.

Convencion: los campos privados llevan guion bajo al inicio (_salarioBase) para distinguirlos visualmente de las propiedades.

Constructor

El constructor: inicializar el objeto en el momento de crearlo

Analogia: Cuando RRHH da de alta a un nuevo empleado, completa el formulario en ese mismo momento: nombre, puesto y salario. No existe una ficha de empleado "vacia que se llena despues". El constructor hace lo mismo: recibe los datos en el momento del new y los asigna de una vez, garantizando que el objeto nace completo.

class Empleado { private double _salarioBase; private int _horas; public string Nombre; // Constructor: mismo nombre que la clase, sin tipo de retorno public Empleado(string nombre, double salarioBase, int horas) { Nombre = nombre; _salarioBase = salarioBase; _horas = horas; } public double CalcularPago() { return _salarioBase * _horas; } } class Program { static void Main() { // Los datos van directo en el new — el objeto nace completo Empleado e1 = new Empleado("Ana", 12000, 160); Empleado e2 = new Empleado("Luis", 15000, 140); Console.WriteLine(e1.CalcularPago()); // 1920000 Console.WriteLine(e2.CalcularPago()); // 2100000 } }

Miembros estaticos

Instancia vs estatico: el apodo propio vs el apellido familiar

Analogia: Cada empleado de la empresa tiene su propio numero de gafete — ese numero es de instancia: es personal, nadie mas lo tiene. Pero el nombre de la empresa impreso en todos los gafetes es de toda la organizacion: es estatico, compartido por cada empleado contratado, y lo accedes por el nombre de la empresa directamente, no por cada persona.

class Empleado { // Estatico: el "apellido" compartido por todos los objetos public static int TotalEmpleados = 0; public string Nombre { get; private set; } public Empleado(string nombre) { Nombre = nombre; TotalEmpleados++; // cada nuevo objeto actualiza el contador familiar } } class Program { static void Main() { Empleado a = new Empleado("Ana"); Empleado b = new Empleado("Luis"); Empleado c = new Empleado("Mia"); // Se accede por el nombre de la clase, no por el objeto Console.WriteLine(Empleado.TotalEmpleados); // 3 // a.Nombre es "Ana" — apodo personal (instancia) // Empleado.TotalEmpleados es 3 — apellido familiar (estatico) } }

Buenas practicas

Sobreescribir ToString: la etiqueta en la carpeta del expediente

Analogia: En la oficina tienes tres carpetas apiladas sin rotulo. Al abrirlas no sabes si es el contrato de Ana, el expediente de Luis o la auditoria de nomina. Si cada carpeta tiene una etiqueta con el nombre y el departamento, de un vistazo sabes exactamente que hay adentro. ToString() es la etiqueta que le pones al objeto para que al mostrarlo en pantalla diga algo util — no el nombre tecnico de la carpeta.

class Empleado { private string _nombre; private double _salarioBase; private int _horas; public string Nombre { get { return _nombre; } } public double SalarioBase { get { return _salarioBase; } } public int Horas { get { return _horas; } } public Empleado(string nombre, double salarioBase, int horas) { _nombre = nombre; _salarioBase = salarioBase; _horas = horas; } public double CalcularPago() { return _salarioBase * _horas; } // Sin override: Console.WriteLine(e) imprime "Empleado" — carpeta sin rotulo // Con override: imprime la etiqueta util que nosotros definimos public override string ToString() { return $"[{_nombre}] pago mensual: {CalcularPago():C}"; } } class Program { static void Main() { Empleado e1 = new Empleado("Ana", 12000, 160); Empleado e2 = new Empleado("Luis", 15000, 140); Console.WriteLine(e1); // [Ana] pago mensual: $1,920,000.00 Console.WriteLine(e2); // [Luis] pago mensual: $2,100,000.00 } }

Una propiedad con solo get es de solo lectura desde fuera: cualquiera puede leer el valor, pero nadie puede cambiarlo directamente — solo el constructor o los metodos de la clase pueden escribir en el campo privado.

Colecciones de objetos

Arreglo de objetos: la lista de nomina del departamento

using System; class Empleado { public string Nombre { get; private set; } public double SalarioBase { get; private set; } public int Horas { get; private set; } public Empleado(string nombre, double salarioBase, int horas) { Nombre = nombre; SalarioBase = salarioBase; Horas = horas; } public double CalcularPago() => SalarioBase * Horas; public override string ToString() => $"{Nombre,-12} | {CalcularPago(),14:C}"; } class Program { static void Main() { // La nomina: tres empleados distintos, todos del mismo formulario Empleado[] equipo = { new Empleado("Ana", 12000, 160), new Empleado("Luis", 15000, 140), new Empleado("Mia", 11000, 168), }; double totalNomina = 0; foreach (Empleado e in equipo) { Console.WriteLine(e); // usa ToString() totalNomina += e.CalcularPago(); } Console.WriteLine($"\nTotal nomina: {totalNomina:C}"); } } // Ana | $1,920,000.00 // Luis | $2,100,000.00 // Mia | $1,848,000.00 // Total nomina: $5,868,000.00

Organizacion del codigo

Cada clase en su propio archivo: un cuarto para cada cosa

Estructura recomendada

// ── Empleado.cs ────────────────────── class Empleado { public string Nombre { get; private set; } public double SalarioBase { get; private set; } public int Horas { get; private set; } public Empleado(string n, double s, int h) { Nombre = n; SalarioBase = s; Horas = h; } public double CalcularPago() => SalarioBase * Horas; public override string ToString() => $"{Nombre,-12} | {CalcularPago():C}"; } // ── Program.cs ─────────────────────── class Program { static void Main() { // Solo orquestacion y entrada/salida Empleado e = new Empleado("Ana", 12000, 160); Console.WriteLine(e); } }

Por que separar — la analogia del departamento

En la empresa, las reglas de negocio (como se calcula el pago) viven en el departamento de nomina. La pantalla que ve el usuario (como se presenta) es responsabilidad de sistemas. Si cambia la formula de pago, solo se modifica nomina. Si cambia la presentacion, solo se toca la interfaz. Nadie confunde los dos mundos.

  • Cambias la regla de pago → solo editas Empleado.cs.
  • Cambias como se muestra → solo editas Program.cs.
  • Empleado.cs puede reutilizarse en otro proyecto sin tocar nada mas.

Ejemplo integrador

Clase CuentaBancaria: encapsulamiento en accion

class CuentaBancaria { private double _saldo; public string Titular { get; private set; } public double Saldo => _saldo; // ventanilla de solo lectura public CuentaBancaria(string titular, double saldoInicial) { if (saldoInicial < 0) throw new ArgumentException("Saldo inicial no puede ser negativo"); Titular = titular; _saldo = saldoInicial; } public void Depositar(double monto) { if (monto <= 0) throw new ArgumentException("Monto invalido"); _saldo += monto; } public void Retirar(double monto) { if (monto <= 0 || monto > _saldo) throw new ArgumentException("Retiro no valido"); _saldo -= monto; } public override string ToString() => $"[{Titular}] saldo actual: {_saldo:C}"; } class Program { static void Main() { CuentaBancaria c = new CuentaBancaria("Ana", 500000); c.Depositar(200000); // _saldo = 700 000 c.Retirar(100000); // _saldo = 600 000 Console.WriteLine(c); // [Ana] saldo actual: $600,000.00 // c._saldo = 9999; // ERROR: campo privado, no accesible } }

Modificadores de acceso

public, private, protected: las zonas de acceso de la empresa

Analogia: public es la recepcion — cualquier visitante entra. private es la boveda de RRHH bajo llave — solo el personal autorizado entra, nadie externo. protected es la sala de reunion interna — los empleados del departamento (subclases) tienen acceso, pero los visitantes externos no. Lo veremos en herencia, sesion 25.

ModificadorQuien puede accederUso tipico
publicCualquier codigo, desde cualquier archivo.Propiedades y metodos que forman la interfaz visible del objeto.
privateUnicamente el codigo dentro de la misma clase.Campos internos y metodos auxiliares que no deben exponerse.
protectedLa clase y sus clases hijas (subclases).Campos que las subclases necesitan pero el mundo exterior no debe tocar.

Regla de oro: empieza siempre con private. Solo lo abres a public cuando hay una razon clara para que codigo externo lo use.

Solucion practica

Clase Producto: implementacion y uso completos

class Producto { private string _nombre; private double _precio; private int _stock; public string Nombre => _nombre; public double Precio => _precio; public int Stock => _stock; public Producto(string nombre, double precio, int stock) { if (precio <= 0) throw new ArgumentException("Precio invalido"); if (stock < 0) throw new ArgumentException("Stock invalido"); _nombre = nombre; _precio = precio; _stock = stock; } public void Vender(int cantidad) { if (cantidad <= 0 || cantidad > _stock) throw new InvalidOperationException("Stock insuficiente"); _stock -= cantidad; } public void Reabastecer(int cantidad) { if (cantidad <= 0) throw new ArgumentException("Cantidad invalida"); _stock += cantidad; } public override string ToString() => $"{_nombre,-15} | precio: {_precio,10:C} | stock: {_stock,4}"; } class Program { static void Main() { Producto p1 = new Producto("Laptop", 3500000, 10); Producto p2 = new Producto("Teclado", 85000, 50); Producto p3 = new Producto("Mouse", 45000, 30); p1.Vender(3); p2.Vender(12); p3.Reabastecer(20); Console.WriteLine(p1); // Laptop | precio: $3,500,000 | stock: 7 Console.WriteLine(p2); // Teclado | precio: $85,000 | stock: 38 Console.WriteLine(p3); // Mouse | precio: $45,000 | stock: 50 } }

Cierre de sesion

Lo que construiste hoy

Conceptos nuevos

  • Clase: el formulario de contratacion — define la estructura de todos los objetos de ese tipo.
  • Objeto: la ficha del empleado contratado — instancia real creada con new.
  • Encapsulamiento: campos privados + propiedades que validan antes de guardar.
  • Constructor: checklist de alta que garantiza que el objeto entra al sistema en estado valido.
  • Estatico: el nombre de la empresa en el gafete — compartido por todos los objetos.

Que viene en sesion 25

  • Herencia: el hijo recibe todo del padre y agrega lo suyo.
  • Polimorfismo: el mismo mensaje, respuestas distintas segun el tipo.
  • Clases abstractas: formularios que obligan a completar ciertos campos — no generan empleados directamente.
  • Interfaces: contratos de comportamiento que cualquier clase puede firmar.
