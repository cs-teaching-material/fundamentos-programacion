Secuencia, variables, tipos de datos y operadores

Conceptos clave

  • Variable: espacio en memoria con nombre, tipo y valor. Debe declararse antes de usarse.
  • Tipos primitivos: int, double, string, bool.
  • Operadores: aritmeticos (+, -, *, /, %), relacionales (==, !=, <, >) y logicos (&&, ||, !).
  • Modelo IPO: Entrada → Proceso → Salida. Todo programa lo sigue.
// Declaracion y uso int visitas = 18; double precioBase = 100_000.0; string tipo = "premium"; bool tieneDescuento; // Calculo tieneDescuento = visitas > 15 || tipo == "premium"; double factor = tieneDescuento ? 0.80 : 1.0; double precioFinal = precioBase * factor; Console.WriteLine($"Precio: {precioFinal:C}");

Estructuras condicionales: if / else / switch

// Condicional simple if (visitas > 15) descuento = 0.10; // Condicional compuesta if (tipo == "premium") descuento = 0.20; else if (visitas > 15) descuento = 0.10; else descuento = 0.0; // Switch (C# moderno) double tasa = tipo switch { "premium" => 0.20, "basica" => visitas > 15 ? 0.10 : 0.0, _ => 0.0 };

Cuando usar cada uno

  • switch: multiples valores discretos de una misma variable; mas legible que cadenas de if-else.

Repaso U2

Ciclos: for, while, do-while

for — repeticiones contadas

for (int i = 1; i <= n; i++) { // bloque ejecutado n veces }

Usar cuando se conoce el numero de iteraciones de antemano.

while — condicion al inicio

while (condicion) { // puede no ejecutarse }

Usar cuando la condicion puede ser falsa desde el principio.

do-while — al menos una vez

do { // se ejecuta minimo 1 vez } while (condicion);

Ideal para menus y lecturas con validacion: siempre se debe leer al menos una vez antes de verificar.

Repaso U2

Estructuras de almacenamiento: arreglo vs List<T>

Arreglo (tamaño fijo)

int[] visitas = new int[n]; visitas[0] = 18; visitas[1] = 8; // Recorrer con for for (int i = 0; i < visitas.Length; i++) Console.WriteLine(visitas[i]);

Usar cuando el tamaño se conoce de antemano y no cambia.

List<T> (tamaño dinamico)

var nombres = new List<string>(); nombres.Add("Ana"); nombres.Add("Luis"); // Recorrer con foreach foreach (string n in nombres) Console.WriteLine(n);

Usar cuando el numero de elementos no se conoce o crece en ejecucion.

Criterio de eleccion: si se lee N antes del ciclo y N no cambia, un arreglo es suficiente. Si los datos se acumulan sin saber cuantos llegaran, List<T> es mas flexible.

Repaso U3

Funcion: firma, parametros y retorno

Anatomia de una funcion

// [modificador] [retorno] Nombre([params]) static double CalcularDescuento( int visitas, string tipo) { if (tipo == "premium") return 0.20; if (visitas > 15) return 0.10; return 0.0; }

El nombre expresa que hace. El tipo de retorno indica que devuelve.

Mecanismos de paso

  • Por valor (default): la funcion recibe una copia. No modifica el original.
  • out: retornar multiples valores. El caller los recibe como variables de salida.
  • ref: pasar por referencia. La funcion puede modificar el original. Requiere justificacion.

Preferir retorno simple y out sobre ref. Nunca usar globales para pasar resultados entre funciones.

Repaso U3

Jerarquia modular y responsabilidad unica

Niveles de la jerarquia

  • Orquestacion (Main / ProcesarN): coordina el flujo; no contiene logica de negocio.
  • Proceso (CalcularX): aplica reglas; no hace entrada/salida.
  • Helpers (LeerX, MostrarX): gestionan entrada o salida; no calculan.

Regla: cada funcion debe caber en pantalla sin scroll (<20 lineas). Si no cabe, hay que extraer subfunciones.

Principio de responsabilidad unica

Una funcion hace una sola cosa. Si el nombre necesita la palabra "y" para describirse (ej. "leer y calcular"), la funcion tiene al menos dos responsabilidades.

Beneficio clave: si las reglas de negocio cambian, solo se modifica una funcion. El resto no se toca.

Sobrecarga de metodos: el mismo nombre, distintas firmas

// Version 1: horas normales static double CalcularSalarioBruto( double horas, double tarifa) { return horas * tarifa; } // Version 2: con horas extra (recargo x1.5) static double CalcularSalarioBruto( double horasNormales, double horasExtra, double tarifa) { double base = horasNormales * tarifa; double extra = horasExtra * tarifa * 1.5; return base + extra; } // El compilador elige segun los argumentos: CalcularSalarioBruto(40, 15000); // version 1 CalcularSalarioBruto(40, 8, 15000); // version 2

Reglas de la sobrecarga

  • Diferenciacion valida: distinta cantidad o tipo de parametros.
  • No es sobrecarga: solo distinto nombre de parametro o distinto tipo de retorno.
  • El compilador resuelve en tiempo de compilacion; no hay ambiguedad en ejecucion.

Cuando usarla

  • Las versiones realizan la misma operacion conceptual con diferente informacion disponible.
  • El nombre es igualmente valido para todas las versiones.

Sobrecarga vs parametros con valor por defecto

Parametro con valor por defecto

// Una sola firma, parametro opcional static double CalcularSalarioBruto( double horas, double tarifa, double horasExtra = 0) { return horas * tarifa + horasExtra * tarifa * 1.5; } // Ambas llamadas son validas: CalcularSalarioBruto(40, 15000); CalcularSalarioBruto(40, 15000, 8);

Una sola firma. Si no se pasa horasExtra, vale 0.

Cuando preferir cada opcion

  • Valor por defecto: la logica interna es identica; el parametro solo agrega datos. Menos duplicacion.
  • Sobrecarga: la logica difiere segun los datos, o el valor por defecto es semanticamente confuso.

"0 horas extra" y "no aplica horas extra" son conceptos distintos en el dominio laboral. La sobrecarga los separa con mayor precision.

Regla de oro: si al leer la llamada no queda claro cual version se invoca, la sobrecarga no deberia existir.

Problema integrador

Enunciado del gimnasio

Contexto: Un gimnasio registra la asistencia de sus clientes durante un mes. Cada cliente tiene nombre, numero de visitas y tipo de membresia (basica, premium o vip). El programa debe:

  • Leer datos de N clientes (validar que N > 0).
  • Calcular el descuento mensual: basica con >15 visitas → 10%; premium → 20%; vip → calcular con sobrecarga que incluye bonificacion adicional.
  • Determinar el precio final: CalcularPrecioFinal debe tener una sobrecarga con descuento adicional por cliente frecuente.
  • Mostrar al final: cantidad de clientes con descuento, total recaudado y cliente con mas visitas.

Este enunciado combina U1 (variables, condicionales), U2 (ciclo for, acumuladores) y U3 (funciones, sobrecarga, valor por defecto, jerarquia).

Lectura del enunciado

Identificar las partes antes de escribir codigo

¿Que datos entran?

  • N (cantidad de clientes, debe ser > 0).
  • Por cada cliente: nombre, visitas, tipo de membresia.
  • Precio base (constante del programa).

¿Que debe salir?

  • Precio final por cliente (en pantalla al leerlo).
  • Total de clientes con descuento aplicado.
  • Total recaudado.
  • Nombre del cliente con mas visitas.

Condiciones del enunciado: basica+>15v→10%; premium→20%; vip→sobrecarga con bonificacion; frecuente (>20v)→descuento adicional sobre precio final.

Paso 2: CalcularDescuento — logica de negocio aislada

static double CalcularDescuento( int visitas, string tipo) { return tipo switch { "vip" => 0.30, "premium" => 0.20, "basica" => visitas > 15 ? 0.10 : 0.0, _ => 0.0 }; } // Casos: // CalcularDescuento(8, "premium") → 0.20 // CalcularDescuento(18, "basica") → 0.10 // CalcularDescuento(5, "basica") → 0.00 // CalcularDescuento(10, "vip") → 0.30

Decision de diseño

Las reglas de descuento estan en una sola funcion. Si las reglas cambian, solo esta funcion se modifica; el ciclo principal no se toca.

Se retorna el porcentaje (0.20), no el monto en pesos. Eso hace la funcion independiente del precio base.

Paso 3: CalcularPrecioFinal — sobrecarga para cliente frecuente

// Version 1: sin bonificacion adicional static double CalcularPrecioFinal( double precioBase, double descuento) { return precioBase * (1 - descuento); } // Version 2: cliente frecuente (>20 visitas) // descuentoAdicional tiene valor por defecto static double CalcularPrecioFinal( double precioBase, double descuento, double descuentoAdicional = 0.05) { double total = descuento + descuentoAdicional; return precioBase * (1 - total); } // En ProcesarClientes: double precio = visitas > 20 ? CalcularPrecioFinal(base, desc, 0.05) : CalcularPrecioFinal(base, desc);

Aqui se aplican las dos tecnicas

  • Sobrecarga: misma operacion (calcular precio), diferente informacion disponible.
  • Valor por defecto: descuentoAdicional = 0.05 evita repetir el valor en cada llamada.
  • El compilador elige la version segun el numero de argumentos.

Si el descuento adicional cambiara a 0.07 para el proximo mes, solo se modifica este valor por defecto.

static void ProcesarClientes(int n, double precioBase) { int conDescuento = 0; double totalRecaudado = 0; int maxVisitas = 0; string clienteTop = ""; for (int i = 1; i <= n; i++) { LeerCliente(i, out string nom, out int vis, out string tipo); double desc = CalcularDescuento(vis, tipo); double precio = vis > 20 ? Decision de diseño

Este ciclo conoce el flujo y los acumuladores, pero no las formulas ni la presentacion.

Si se necesita cambiar "como se muestra el resumen", no se toca este ciclo.

Ciclo de vida del programa

De la idea a la solucion verificada

1. Lectura del enunciado: identificar entradas, salidas y condiciones. Sin esto no se diseña.

Razonamiento tecnico

Ejercicio 1: elegir la estructura de iteracion correcta

Enunciado: "El programa debe repetir la lectura de datos hasta que el usuario ingrese 'salir'. ¿Que estructura es mas adecuada y por que?"

Analisis comparativo

  • for: repeticiones con numero conocido de veces.
  • while: condicion verificada antes de ejecutar; puede no entrar.
  • do-while: al menos una ejecucion antes de verificar la condicion.

Respuesta justificada

do-while: porque siempre se debe leer al menos una vez antes de poder evaluar si el usuario ingreso 'salir'. Con while, la condicion se verificaria antes de leer, lo que requiere una inicializacion artificial de la variable.

Sesion 22 · Razonamiento 125 / 30
Razonamiento tecnico

Ejercicio 2: predecir la salida de un fragmento

static int Doble(int x) { return x * 2; } static int Suma(int a, int b) { return a + b; } static void Main() { int x = 3; int y = Doble(x); int z = Suma(y, Doble(2)); Console.WriteLine(z); }

Traza de ejecucion

  • x = 3
  • Doble(3) → 6 → y = 6
  • Doble(2) → 4
  • Suma(6, 4) → 10 → z = 10
  • Salida: 10

Seguir el flujo de datos entre funciones mentalmente, sin ejecutar, es una habilidad clave del razonamiento algoritmico.

Ejercicio 3: sobrecarga, valor por defecto y alcance

static double Precio(double base, double desc) => base * (1 - desc); static double Precio(double base, double desc, double extra = 0.05) => base * (1 - desc - extra); static void Main() { double a = Precio(100, 0.20); double b = Precio(100, 0.20, 0.10); double c = Precio(100, 0.20, 0); Console.WriteLine($"{a} {b} {c}"); }

Analisis

  • a: 2 args → version 1 → 100 × 0.80 = 80
  • b: 3 args, extra=0.10 → version 2 → 100 × 0.70 = 70
  • c: 3 args, extra=0 → version 2 → 100 × 0.80 = 80
  • Salida: 80 70 80

Notar que a y c producen el mismo resultado por distintos caminos: version 1 vs version 2 con extra=0.
