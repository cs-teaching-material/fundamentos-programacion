Problema del codigo monolitico

Todo en Main() = alto costo de mantenimiento

// Monolitico: entrada, calculo, validacion y salida en un solo bloque static void Main() { Console.Write("Nombre: "); string nombre = Console.ReadLine(); Console.Write("Nota 1: "); double n1 = double.Parse(Console.ReadLine()); Console.Write("Nota 2: "); double n2 = double.Parse(Console.ReadLine()); Console.Write("Nota 3: "); double n3 = double.Parse(Console.ReadLine()); double prom = (n1 + n2 + n3) / 3; Console.WriteLine($"Estudiante: {nombre}"); Console.WriteLine($"Promedio: {prom}"); Console.WriteLine(prom >= 3.0 ? "Aprueba" : "No aprueba"); }

Si falla una parte, revisar todo el bloque toma mas tiempo y aumenta errores.

Principio DRY

Idea central

DRY = Dont Repeat Yourself. Evita repetir logica igual en distintas partes del programa.

  • Menos codigo duplicado.
  • Menos puntos de error.
  • Cambios mas rapidos y seguros.

Patron recomendado

static double CalcularPromedio(double a, double b, double c) { return (a + b + c) / 3; } static string Estado(double promedio) { return promedio >= 3.0 ? "Aprueba" : "No aprueba"; }
Diseño con reuso real 5 / 16
Concepto de funcion, firma y contrato

Estructura de una funcion

static double CalcularPromedio(double n1, double n2, double n3) { return (n1 + n2 + n3) / 3; }
  • Firma: nombre + parametros + tipo de retorno.
  • Contrato: que recibe y que garantiza devolver.

Checklist de contrato

  • Parametros validos y en orden correcto.
  • Retorno coherente con el problema.
  • Sin efectos colaterales innecesarios.
  • Nombre descriptivo de la accion.

Parametros y retorno: flujo de datos

Regla didactica: una funcion no "adivina" datos; solo trabaja con parametros y devuelve un resultado definido.

Comprension de entrada/salida en funciones 7 / 16
Separacion UI y logica

Capa UI (interaccion)

  • Leer datos con Console.ReadLine().
  • Mostrar mensajes al usuario.
  • No mezclar formulas complejas aqui.

Capa logica (proceso)

  • Funciones de calculo y validacion.
  • Sin dependencias de pantalla.
  • Facil de probar con casos de prueba.

Beneficio: si cambia la interfaz, la logica puede mantenerse intacta.

Comparativo de codigo - parte 4 (vision integral)

Antes: flujo monolitico

Main() ├─ Lee nombre ├─ Lee nota 1 ├─ Lee nota 2 ├─ Lee nota 3 ├─ Calcula promedio ├─ Decide estado └─ Imprime reporte Problema: - Todo depende de un solo bloque. - Un cambio pequeno obliga a tocar Main().

Despues: flujo modular

Main() ├─ LeerNota() ├─ CalcularPromedio() ├─ ObtenerEstado() └─ MostrarReporte() Ventaja: - Cada funcion tiene una responsabilidad. - Los cambios se hacen en puntos localizados. - El codigo se prueba por partes.

Idea final: modularizar no cambia solo la forma del codigo, cambia la estrategia de mantenimiento y depuracion del programa.

Practica guiada en clase

Construir calculadora de promedio modular

Objetivo del ejemplo: desarrollar paso a paso un programa en C# que solicite nombre y tres notas, calcule promedio con una funcion, determine estado con otra funcion y muestre reporte final.

  • Funcion 1: CalcularPromedio(double,double,double).
  • Funcion 2: ObtenerEstado(double).
  • Funcion 3: MostrarReporte(string,double,string).
  • Validar que cada nota este entre 0 y 5.

Trabajo en acompanamiento docente: primero se construye en plenaria y luego cada estudiante lo replica con variaciones.

1. Diseno Define firmas y orden de llamadas. Propone nombres de funciones y parametros.

3. Pruebas Ejecuta casos de prueba en vivo. Compara resultados y ajusta errores de logica.
4. Variacion Plantea un cambio en regla de aprobacion. Modifica solo la funcion afectada (DRY).
