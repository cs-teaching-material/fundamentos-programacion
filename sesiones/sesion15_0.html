Todo en Main: el diagnostico

// Programa tipico del final de Unidad 2 static void Main() { var lista = new List<double>(); string opcion; do { Console.Write("1-Agregar 2-Reporte 0-Salir: "); opcion = Console.ReadLine(); if (opcion == "1") { Console.Write("Valor: "); double v = double.Parse(Console.ReadLine()); if (v < 0) { Console.WriteLine("Invalido"); } else lista.Add(v); } if (opcion == "2") { double sum = 0; foreach (var x in lista) sum += x; // ... 30 lineas mas ... } } while (opcion != "0"); }

Senales de alarma

  • Validacion de entrada repetida en varios if
  • Calculo del promedio aparece dos veces
  • Imposible probar el reporte sin ejecutar todo el menu
  • Un cambio en el formato de salida toca tres lugares

Concepto A — Principio DRY

Don't Repeat Yourself

✗ Viola DRY

// Validacion repetida en dos opciones del menu if (opcion == "1") { double v = double.Parse(Console.ReadLine()); if (v < 0 || v > 10) Console.WriteLine("Fuera de rango"); } if (opcion == "3") { double v = double.Parse(Console.ReadLine()); if (v < 0 || v > 10) // duplicado Console.WriteLine("Fuera de rango"); }

✓ Aplica DRY

// Una sola funcion valida en ambos casos static bool EsValida(double v) { return v >= 0 && v <= 10; } // En el menu: if (!EsValida(v)) Console.WriteLine("Fuera de rango");

Si el rango cambia a 0-20, modificas un solo lugar.

Las cuatro partes de una firma en C#

ParteQue significaEjemplo en la firma
ModificadorDefine alcance y contexto de uso del metodo.static
Tipo de retornoIndica que dato devuelve al terminar.double

Regla: el nombre empieza con verbo (Calcular, Validar, Mostrar, Obtener) y describe exactamente lo que hace. Jamas Proceso1.

Concepto B — Firma y contrato

Firma: forma tecnica

static double CalcularPromedio(double n1, double n2, double n3) { return (n1 + n2 + n3) / 3; }
  • Nombre: describe la accion.
  • Parametros: datos minimos para resolver.
  • Retorno: resultado esperado del proceso.

Contrato: acuerdo de uso

  • Recibe 3 notas validas en rango definido por el curso.
  • Devuelve un promedio numerico en tipo double.
  • No imprime en pantalla ni solicita datos.
  • Siempre produce el mismo resultado para la misma entrada.

Idea clave: cuando la firma y el contrato estan claros, el trabajo en equipo es mas facil porque cada funcion se puede usar sin adivinar su comportamiento.

Sesion 15 · Firma y contrato 8 / 21
Concepto B — Separacion UI / logica

La regla de oro: una funcion no "adivina" datos

✗ Mezclado

static double CalcularPromedio() { // La funcion lee Y calcula: mezcla UI y logica Console.Write("Nota 1: "); double a = double.Parse(Console.ReadLine()); Console.Write("Nota 2: "); double b = double.Parse(Console.ReadLine()); return (a + b) / 2; }

✓ Separado

// Logica pura: recibe datos, devuelve resultado static double CalcularPromedio(double a, double b) { return (a + b) / 2; } // Main se encarga de la UI: Console.Write("Nota 1: "); double n1 = double.Parse(Console.ReadLine()); Console.Write("Nota 2: "); double n2 = double.Parse(Console.ReadLine()); double p = CalcularPromedio(n1, n2);

Beneficio: si cambias la interfaz (web, archivo, API), la logica de calculo no se toca.

Practica guiada — Paso 3: variacion DRY

Cambio de regla: aprobacion en 3.5

Cambio solicitado: la nota minima de aprobacion pasa de 3.0 a 3.5.
En codigo monolitico: debes buscar y editar en todos los if donde aparece 3.0. Riesgo de olvidar alguno.
En codigo modular (DRY): modificas unicamente ObtenerEstado. Un lugar. Sin riesgo de inconsistencia.
// Solo esto cambia (forma tradicional recomendada): static string ObtenerEstado(double prom) { if (prom >= 3.5) { return "Aprobado"; } return "Reprobado"; } // Referencia unica de azucar sintactico (equivale a lo anterior): // static string ObtenerEstado(double prom) => prom >= 3.5 ? "Aprobado" : "Reprobado";

Practica guiada — Paso 4: descomposicion

Metodo en cuatro pasos para modularizar

Paso 1: definir entradas y salidas del problema.
Paso 2: separar subproblemas en funciones pequenas y nombrarlas con verbos.
Paso 3: conectar funciones desde Main sin mezclar logica de negocio.
Paso 4: probar cada funcion con casos controlados antes de integrar.

Este metodo evita saltar directo al codigo y reduce retrabajo durante la implementacion.

Practica guiada en clase

Refactorizar un programa monolitico

Enunciado: Trabajaremos en conjunto sobre un programa monolitico (todo en Main) que calcula area y perimetro de un rectangulo y determina si es cuadrado. El objetivo es extraer tres funciones con firma correcta, manteniendo la logica original.

  • Funcion 1: calcula el area (retorna double).
  • Funcion 2: calcula el perimetro (retorna double).
  • Funcion 3: determina si es cuadrado (retorna bool).
  • Main solo lee datos, llama funciones y muestra resultados.

Modalidad: solucion en plenaria, despues cada estudiante replica la version final en su entorno para verificar comprension.

static double CalcularArea(double baseRect, double alturaRect) { return baseRect * alturaRect; } static double CalcularPerimetro(double baseRect, double alturaRect) { return 2 * (baseRect + alturaRect); } static bool EsCuadrado(double baseRect, double alturaRect) { return baseRect == alturaRect; } static void Main() { Console.Write("Base : "); double baseRect = double.Parse(Console.ReadLine()); Console.Write("Altura: "); double alturaRect = double.Parse(Console.ReadLine()); double area = CalcularArea(baseRect, alturaRect); double perimetro = CalcularPerimetro(baseRect, alturaRect); bool esCuadrado = EsCuadrado(baseRect, alturaRect); Console.WriteLine($"Area : {area}"); Console.WriteLine($"Perimetro : {perimetro}"); Console.WriteLine($"Es cuadrado: {esCuadrado}"); }
