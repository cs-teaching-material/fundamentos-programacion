Proposito de la sesion

Dar el salto: de caso unico a procesamiento por lotes

  • Implementar una solucion modular que procese varios empleados en una misma ejecucion.
  • Registrar fallos por cada registro sin detener todo el lote.
  • Construir acumuladores del proceso: total neto, total pago y tasa de error.
  • Emitir un resumen consolidado al finalizar.
  • Dejar base lista para refinamiento de calidad en sesion 21.

Problema real

Enunciado exacto del problema

Enunciado: Construir un programa en C# que procese la nomina de N empleados en una sola ejecucion; cada registro debe validarse de forma independiente, los errores deben reportarse sin detener el proceso general y, al finalizar, se debe mostrar un resumen consolidado con exitos, errores, total neto, total pagado y tasa de error.

Condiciones clave

  • Si un registro falla, los demas se siguen procesando.
  • Cada error queda asociado al indice del registro.
  • El resumen final representa todo el lote.

Resultado esperado

Un flujo robusto de procesamiento por lotes, trazable y con salida final clara para su analisis en clase.

Decision de diseno

De donde sale ProcesarLote?

Idea

Si el problema es "procesar muchos", necesitamos una funcion que coordine el ciclo completo.

A esa funcion le ponemos un nombre intencional: ProcesarLote(n).

static void Main() { int n = LeerCantidad(); ProcesarLote(n); } // Nace para resolver el problema de muchos registros static void ProcesarLote(int n) { for (int i = 1; i <= n; i++) { // procesar registro i } }
Sesion 20 · Decision4 / 20
Diseno modular

Que hace exactamente ProcesarLote

Responsabilidad central

  • Recibe N registros a procesar.
  • Recorre cada registro (for).
  • Decide si suma a exitos o errores.
  • Al final llama al consolidado.

Que NO hace

  • No calcula formulas de negocio detalladas.
  • No valida todos los campos por si sola.
  • No imprime cada detalle de dominio.
  • No reemplaza a TryProcesarEmpleado.

Resumen simple: ProcesarLote coordina el proceso; las otras funciones resuelven tareas especificas.

Diseno modular

Patron Try por registro (sin romper el lote)

static bool TryProcesarEmpleado( int idx, out double neto, out double pago, out string error) { neto = 0; pago = 0; error = ""; if (!LeerEntradaRegistro(idx, out double h, out double t)) { error = "Entrada invalida"; return false; } if (!ValidarReglas(h, t, out error)) return false; neto = CalcularNeto(h, t); pago = CalcularPagoFinal(neto); return true; }

Clave didactica

  • Una falla no detiene todo el proceso.
  • Se conserva resultado parcial del lote.
  • El error queda asociado a indice de registro.
  • Habilita depuracion basada en datos.

Implementacion

Acumuladores del proceso

Estado permitido

  • int exitos
  • int errores
  • double totalNeto
  • double totalPago

Son datos del proceso por lotes, no del calculo de cada empleado.

Reglas

  • Solo ProcesarLote actualiza acumuladores.
  • Calculos unitarios siguen puros por parametro.
  • MostrarConsolidado no recalcula formulas.
  • Bitacora guarda errores por registro.

Paso 2 · ProcesarLote(): problema -> un error podria detener todo el ciclo.

En cada paso implementamos una solucion puntual antes de avanzar al siguiente bloque.

Practica guiada

Paso 1: Main() define el punto de entrada

Problema

Sin una cantidad valida, no existe proceso de lote confiable.

Solucion

Validar n antes de llamar a ProcesarLote(n).

static void Main() { Console.Write("Cantidad de registros: "); if (!int.TryParse(Console.ReadLine(), out int n) || n <= 0) { Console.WriteLine("Cantidad invalida"); return; } ProcesarLote(n); }

Paso 2: ProcesarLote() coordina sin detenerse

Problema

Si un registro falla, se puede cortar todo el proceso y perder resultados validos.

Solucion

Controlar cada registro con if/else, acumular exitos y registrar errores.

static void ProcesarLote(int n) { int exitos = 0; int errores = 0; double totalNeto = 0; double totalPago = 0; for (int i = 1; i <= n; i++) { if (TryProcesarEmpleado(i, out double neto, out double pago, out string err)) { exitos++; totalNeto += neto; totalPago += pago; } else { errores++; RegistrarError(i, err); } } }

Paso 3: TryProcesarEmpleado() aisla fallas

Problema

Entradas no numericas o reglas invalidas pueden contaminar el calculo.

Solucion

Usar salidas anticipadas con return false y mensaje de error por registro.

if (!LeerEntradaRegistro(idx, out double horas, out double tarifa)) { error = "Entrada no numerica"; return false; } if (!ValidarReglas(horas, tarifa, out error)) return false; neto = CalcularNeto(horas, tarifa); pago = CalcularPagoFinal(neto); RegistrarLog(idx, "ok"); return true;

Se observa la diferencia: aqui se controla exito/error por registro, no por caso unico.

Paso 4: MostrarConsolidado() cierra el proceso

Problema

Sin consolidado final, los resultados quedan dispersos y no se pueden interpretar.

Solucion

Centralizar resumen de registros, exitos, errores, totales y tasa de error.

static void MostrarConsolidado( int n, int exitos, int errores, double totalNeto, double totalPago) { double tasaError = (double)errores / n; Console.WriteLine("--- RESUMEN LOTE ---"); Console.WriteLine($"Registros: {n}"); Console.WriteLine($"Exitos: {exitos}"); Console.WriteLine($"Errores: {errores}"); Console.WriteLine($"Total neto: {totalNeto}"); Console.WriteLine($"Total pago: {totalPago}"); Console.WriteLine($"Tasa error: {tasaError:P1}"); }

Analisis

Errores frecuentes en lotes

Fallas tipicas

  • Detener el for ante primer error.
  • No inicializar acumuladores.
  • Reusar variable error entre iteraciones.
  • No registrar indice de registro fallido.

Correcciones

  • Usar Try y continuar siguiente i.
  • Inicializar acumuladores antes del ciclo.
  • Resetear out en cada llamada Try.
  • RegistrarError(i, motivo) siempre.
