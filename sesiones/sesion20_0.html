Proposito de la sesion

Dar el salto: de caso unico a procesamiento continuo

  • Implementar una solucion modular robusta que procese multiples empleados en una misma ejecucion (Batch Processing), simulando sistemas en el mundo real.

Problema real

Enunciado exacto del problema a resolver

Mision: Construir un programa en C# que procese la nomina de N cantidad de empleados en una unica sesion de uso. Cada registro se validara de forma aislada e independiente; si hay fallas, deben reportarse en una bitacora (consola) sin detener a los demas. Al culminar, el sistema emitira un reporte detallando exitos, fallos, montos absolutos netos y pagados, junto a su relacion porcentual de errores.

Condiciones innegociables

  • Si un empleado tiene horas negativas (ej. -5), su error se notifica, pero el sistema continua con el siguiente.
  • Todo error debe incluir el "Numero de registro" (Indice i) para rapida ubicacion visual.
  • Los totales finales engloban solo casos validos.

Resultado y valor esperado

Un sistema dotado de tolerancia a fallos de entrada (Resiliencia), donde el usuario experimentara la fluidez de no tener que reiniciar ante el minimo error tipografico. Este es el principio de los "Sistemas Tolerantes a Fallos".

Patron 'Try' por registro individual (Seguridad)

static bool TryProcesarEmpleado( int idx, out double neto, out double pago, out string error) { // Inicializar variables 'out' obligatoriamente neto = 0; pago = 0; error = ""; if (!LeerEntradaRegistro(idx, out double h, out double t)) { error = "Entrada invalida"; return false; } if (!ValidarReglas(h, t, out error)) return false; // Falla por contexto del negocio neto = CalcularNeto(h, t); pago = CalcularPagoFinal(neto); return true; // Logro absoluto }

Por que devolver booleanos?

  • Evita Exception Handlers pesados para simples validaciones de control (Control de Flujo vs Excepciones).
  • Los parametros out extraen informacion de exito sin forzar la creacion de Estructuras complejas aun.
  • Promueve legibilidad natural en los if(Try...) (como hace int.TryParse nativo de C#).

Implementacion del Estado

Acumuladores vivos del proceso iterativo

Variables del Entorno ('Scoped State')

  • int exitos - Incrementa si Try es True.
  • int errores - Incrementa si Try es False.
  • double totalNeto - La hucha donde caen sueldos brutos.
  • double totalPago - Egresos reales netos sumados de la tesoreria.

Su existencia y ciclo vital (Scope) nacen y mueren estrictamente dentro de ProcesarLote, no flotando aislados en la clase principal.

Reglas inquebrantables de mutacion

  • ProcesarLote es la unica entidad autorizada a sumarles `+1` o `+= valor`.
  • Las funciones matematicas base (CalcularNeto) no pueden "tocar" los globales ni recibir referencia de ellos. Debe ser funcional Puro.
  • Si un Try resulta 'False', los acumuladores financieros se protegen e ignoran el ciclo vacio para no contaminarse.

Paso 1: Main() define el arranque y cierra el paso al error

El riesgo latente (Problema)

Si el usuario digita "veinte" o -5 como numero de registros iniciales, el lote entero carece de sentido topologico y detonaría o bucles fantasmas o bloqueos de ejecución completos.

Solucion Restrictiva Anticipada

Atrapar n con int.TryParse evitando excepciones graves formativas. Si el ingreso no supera nuestra expectativa `> 0`, detenemos tajantemente al proceso con un retorno elegante de salida tempana.

static void Main() { Console.Write("Cantidad de registros al lote: "); // Validacion integral de casteo y regla de negocio general if (!int.TryParse(Console.ReadLine(), out int n) || n <= 0) { Console.WriteLine("Advertencia: Cantidad no operante invalida."); return; // El proceso aborta elegantemente (Fail-Fast a nivel raiz) } ProcesarLote(n); }

Paso 2: ProcesarLote() avanza absorbiendo golpes

El caos del ciclo inexperto (Problema)

Tradicionalmente los estudiantes permiten que el programa colapse a la primera excepcion interna que salta en un for. Perdiendo toda meta o estado logrado procesado antes del quiebre.

La Boveda Segura (Solucion)

Un IF que controla al validador Try. Los resultados limpios afectan a los acumuladores positivos, y los Falsos derivan sumas puras al contador de errores, notificando incidencias con RegistrarError() especificando el indice de afectacion.

static void ProcesarLote(int n) { int exitos = 0, errores = 0; double totalNeto = 0, totalPago = 0; for (int i = 1; i <= n; i++) { if (TryProcesarEmpleado(i, out double neto, out double pago, out string err)) { exitos++; totalNeto += neto; totalPago += pago; // Todo Ok! } else { errores++; RegistrarError(i, err); // Aisla e identifica } } }

Paso 3: Aislamiento interno de fallos en el TryProcesar

Falsos Positivos o Contaminacion Global

Procesar entradas de consola vacías o aplicar deducciones matematicas a horas absurdamente irreales en reglas de negocio (e.g. -10 horas), darian netos corrompidos, desestabilizando financieramente los finales.

'Guard Clauses' protectoras

Existen validaciones tempranas donde si `ValidarReglas` reprueba, el metodo salta con `return false` al instante ("Fail-Fast"), dejando los montos `out` intactos en cero, y exportando al padre la causa verbal en el `string error`.

Paso 4: Auditoria transparente via MostrarConsolidado()

El dilema de la caja negra

Cuando un lote voluminoso finaliza de procesar silenciosamente y solo vemos finalizado el ciclo en la consola, se siente desconectado e inseguro para un cajero o auditor operativo.

Solucion de Tabulacion Centralizada

Delegar la carga estética de UI terminal tomando las variables base acumuladas, para luego derivar internamente aquí tasas porcentuales, entregando un reporte humano conciso tabulado libre de lógica de negocios.

static void MostrarConsolidado( int n, int exitos, int errores, double totalNeto, double totalPago) { double tasaError = (double)errores / n; Console.WriteLine(" --- ANALISIS FINANCIERO DE FIN DE LOTE ---"); Console.WriteLine($"Registros evaluados globales: {n}"); Console.WriteLine($"Transacciones confirmadas: {exitos}"); Console.WriteLine($"Inconsistencias halladas: {errores}"); Console.WriteLine($"Acumulado de Brutos (Neto): {totalNeto:C2}"); Console.WriteLine($"Impacto en Egresos final: {totalPago:C2}"); Console.WriteLine($"Tasa desvio del lote (Error): {tasaError:P1}"); }

Tips Core de la industria: Antipatrones a desechar

Errores lógicos de arquitectura comunes al lidiar con flujos pesados ("Loops")

Patologías Críticas de flujo

  • "Break impulsivo": Abortar del todo un `for` al tropezar con el primer error de tipeo sin reportar incidencias ni proseguir analíticamente.
  • Caos Local-Global de variables: Reinicializar erróneamente compiladores (`exitos = 0`) DENTRO del ciclo en lugar de protegerlos AFUERA de la iteracion general maestra del método `Lote`.
  • Variable `out string err` pegajosa: No inicializar o purgar el string receptor en la función Try base, arrastrando mensajes de error de la iteración anterior a registros válidos actuales.

Correcciones Estándar Obligatorias

  • Permitir ciclos limpios que ignoren o utilicen comandos `continue;` controlados absorbiéndose sanos.
  • Restringir modificadores de acumuladores puramente a la entidad contenedora Padre global `ProcesarLote` fuera y alejado previo de su ciclo for central de transacciones globales inmodificadas por agentes u métodos foráneos abstractos delegados unitarios iterativos aislados locales propios unitarios aislados limpios del lote total global inmodificable por funciones externas o validaciones individuales fuera del orquestador.
