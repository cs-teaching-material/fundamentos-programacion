Concepto A — Descomposicion

De un problema grande a varios mas manejables

Idea clave

Descomponer es convertir una meta general en decisiones pequenas que el programa pueda ejecutar en orden.

  • Cada subproblema responde una pregunta concreta.
  • Las funciones nacen de esas preguntas.
  • Un buen diseño reduce las dependencias entre subproblemas.
  • Mientras mas claro sea cada bloque, mas facil sera probar y corregir errores.

Regla practica: si una funcion responde a mas de una pregunta importante del enunciado, probablemente debe dividirse en varias funciones.

Criterio de calidad: una funcion bien definida tiene un objetivo unico, entradas claras y una salida comprensible.

Concepto A — Niveles de funciones

Capas dentro de un programa

Nivel Ejemplo Responsabilidad de programacion
0 — Main Main Orquestar el flujo general: llamar funciones de alto nivel en orden.
1 — Alto nivel RegistrarVenta, ProcesarPedido Traducir el problema al flujo principal: entrada, proceso y salida.
2 — Calculo CalcularSubtotal, CalcularIVA, CalcularDescuento Aplicar reglas de negocio concretas sobre datos ya validados.
3 — Helpers EsMayorista, ValidarCantidad Resolver decisiones simples y reusable helpers que reducen repeticion.

Esta organizacion mejora la abstraccion: cada nivel se concentra en una escala distinta del problema.

Concepto B — Jerarquia de funciones

Arbol de llamadas para un flujo de venta

Cada flecha indica una llamada de una funcion a otra. Las funciones de abajo no deberian saber detalles de la interfaz de usuario. Asi se protege el nucleo logico del programa.

Sesion 18 · Jerarquia 6 / 18
Concepto B — Lectura del arbol

Quien llama a quien en el flujo de venta

Interpretacion

  • Main decide el orden de las funciones de nivel 1.
  • CalcularTotales delega calculos a funciones de nivel 2.
  • IngresarPedido y MostrarResumen se encargan de la interfaz de consola.
  • Las funciones de calculo no dependen de Console.
  • El resultado de una funcion se convierte en entrada de otra (encadenamiento de datos).

Advertencia: si una funcion conoce demasiados detalles de otras capas (por ejemplo, leer y escribir en consola y ademas calcular), la jerarquia se debilita y el programa se vuelve dificil de mantener.

Traduccion conceptual: baja cohesion + alto acoplamiento = codigo fragil.

Concepto C — Patrón de diseño

Cuatro pasos para descomponer un problema de comercio

Paso 1: Subrayar verbos y datos clave en el enunciado (leer, calcular, mostrar, aplicar, etc.).
Paso 2: Agrupar acciones relacionadas (entrada, calculo, reporte) en posibles funciones.
Paso 3: Proponer nombres de funciones para cada grupo, indicando parametros, retorno y que datos consume/produce.
Paso 4: Dibujar el arbol de llamadas: quien llama a quien, en que orden y en que momento se transforma cada dato.

Este patron tambien sirve para problemas de inventario, nomina, reservas o cualquier dominio de software.

Practica guiada — Enunciado

Mini sistema de ventas en una panaderia

Enunciado: La panaderia vende productos con un precio unitario. El sistema debe:

  • Leer producto, cantidad y precio unitario.
  • Calcular subtotal de la venta.
  • Aplicar descuento por cantidad (mayorista).
  • Calcular IVA y total a pagar.
  • Mostrar un resumen claro de la venta.

Por ahora nos enfocamos en el diseño de la solucion: que hace cada parte, como se conecta y que datos intercambia.

Practica guiada — Esqueleto de funciones

Firmas sin implementar (modelo)

static void IngresarPedido() { /* Leer datos de la venta */ } static double CalcularSubtotal(int cantidad, double precioUnitario) { /* ... */ } static double CalcularDescuento(int cantidad, double subtotal) { /* ... */ } static double CalcularIVA(double base) { /* ... */ } static double CalcularTotal(double base, double iva) { /* ... */ } static void MostrarResumen() { /* Mostrar resultados */ } static void RegistrarVenta() { // Llamar a las funciones anteriores en orden logico } static void Main() { RegistrarVenta(); }

Las firmas representan contratos: antes de codificar el cuerpo, define con precision entradas, salidas y responsabilidad.

Sesion 18 · Esqueleto 12 / 18
Analisis de mal diseño

Cuando todo queda en una funcion gigante

static void ProcesarVenta() { // Leer datos Console.Write("Producto: "); string producto = Console.ReadLine(); Console.Write("Cantidad: "); int cantidad = int.Parse(Console.ReadLine()); Console.Write("Precio unitario: "); double precio = double.Parse(Console.ReadLine()); // Calculos mezclados con impresion double subtotal = cantidad * precio; double descuento = cantidad >= 10 ? subtotal * 0.1 : 0; double baseImponible = subtotal - descuento; double iva = baseImponible * 0.19; double total = baseImponible + iva; Console.WriteLine("Total a pagar: " + total); }

Problemas: mezcla entrada, calculos y salida; no se puede reutilizar facilmente y es dificil de probar por partes.

Efecto en fundamentos: cuando no hay descomposicion, se pierde trazabilidad logica y aumenta el riesgo de errores ocultos.

Sesion 18 · Evaluacion 16 / 18
Errores frecuentes

Patrones de mal diseño a evitar

Error Consecuencia Correccion
Muchas responsabilidades en una sola funcion. Codigo dificil de entender y probar. Dividir por tipos de tarea: entrada, calculo, salida.
Funciones de calculo que usan Console. Fuerte acoplamiento con la interfaz de consola. Mover lecturas y escrituras a funciones de nivel 1.
Nombres genericos como HacerTodo. Falta de claridad en el arbol de funciones. Renombrar usando verbos y sustantivos concretos.
Saltos de nivel desordenados. Jerarquia confusa; Main llama funciones demasiado especificas. Introducir funciones de alto nivel que agrupen tareas.
