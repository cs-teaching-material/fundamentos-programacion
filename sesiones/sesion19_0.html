Caso critico

Sombra de variables: ejemplo natural de auditoria

Contexto real: queremos saber cuantas acciones registra el programa. Por eso existe contadorLlamadas como dato compartido sencillo.

static int contadorLlamadas = 0; static void RegistrarAuditoria(string evento) { int contadorLlamadas = 1; // sombra: crea otro contador contadorLlamadas++; Console.WriteLine(evento + " -> #" + contadorLlamadas); }

Que debe entenderse aqui

  • Por que existe el campo: para llevar una cuenta general de acciones del programa.
  • La falla es la sombra: el contador local oculta el compartido.
  • Resultado: el log parece avanzar, pero el contador real nunca cambia.
  • No falla static; falla la regla de uso. Solo la funcion de auditoria debe tocar el campo compartido sin redeclararlo.

Puente a la siguiente slide: definimos reglas simples para usar datos compartidos sin confusion.

Idea de transicion: static tiene sentido cuando el dato se usa en todo el programa; para cada calculo individual, preferimos parametros y retornos.

Decision de estado compartido

Cuando compartir estado vale la pena

Continuidad del caso anterior: si el campo sirve para contar acciones generales, se conserva; si es un dato de cada trabajador, debe ir por parametros.

Uso valido (con regla)

  • Contador de auditoria: una sola funcion lo incrementa.
  • Configuracion global de solo lectura: no cambia en ejecucion.
  • Tarifa por defecto institucional: valor inicial; luego manda la tarifa ingresada y validada.
  • Cache controlada: politica clara de actualizacion e invalidez.

Clave: el dato compartido debe tener un responsable claro.

Uso invalido (genera deuda)

  • Guardar entradas de usuario por comodidad.
  • Pasar resultados entre funciones sin parametros.
  • Acoplar calculos a estado oculto mutable.

Sintoma: "funciona a veces" y cuesta depurar causa raiz.

// Caso correcto: estado compartido minimo static int contadorLlamadas = 0; static void RegistrarAuditoria(string evento) { contadorLlamadas++; Console.WriteLine(evento); } // Dominio sin dependencia oculta static double CalcularNeto(double horas, double tarifa) { return horas * tarifa; }

Checklist de decision

  1. Este dato lo usan varias funciones de verdad?
  2. Es un valor general (por ejemplo, un valor por defecto) y no un dato de cada persona?
  3. Si cambia, hay una sola funcion responsable?
  4. Puedo probar el calculo sin depender de ese dato compartido?
  5. Si la respuesta es "no", usa parametro o retorno.

Principio operativo: compartir pocos datos, que sean visibles y con motivo; el calculo principal debe funcionar con parametros.

Analogia visual

Tablero comun vs cuaderno personal

Tablero comun

Sirve cuando toda la clase necesita mirar el mismo dato.

Cuaderno personal

Ideal para datos temporales que no deben salir de una funcion.

Idea docente: si el dato se parece a un cuaderno personal, debe ser local o parametro. Si se parece a un tablero comun, exige una regla estricta para saber quien lo actualiza.

Errores comunes

Fallas que deben evitarse

  • Declarar horas y tarifa como campos compartidos.
  • Reusar nombres y crear sombra sin advertirlo.
